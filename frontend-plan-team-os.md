# План разработки TeamOS — фронтенд (визуальная часть)

Бэкенда пока нет, поэтому весь план строится на **мок-данных через слой API-клиента**: интерфейсы данных и функции запросов описываются сразу, но возвращают фикстуры. Когда появится бэкенд, меняется только реализация этого слоя — компоненты не трогаем.

**Стек:** React + TypeScript + Vite, Tailwind CSS, TanStack Query (поверх мок-API), Zustand (UI-состояние), ReactUse (@reactuses/core), React Router, TipTap (редактор), dnd-kit (drag-and-drop).

- В мок-API сразу заложите задержку 300–500 мс и случайные ошибки (5%) — иначе интерфейс не будет спроектирован под загрузку и фейлы, и при подключении реального бэкенда всё «поплывёт».
- ReactUse даёт готовые удобные хуки, что сильно ускорит разработку и уменьшит бойлерплейт.

| 1.3 | Карточка должности: описание функций, привязанные регламенты и обязательные курсы (пока ссылки на моки) | low |

| 1.5 | Список сотрудников: таблица с поиском, фильтром по отделу, модалка приглашения (email/ссылка, выбор должности) | medium |

**Нюансы этапа:**
- Визуализация дерева — главный риск. Два пути: горизонтальная орг-диаграмма (красиво, но сложна на больших структурах и мобильных) или вложенный список-дерево (проще и практичнее). Рекомендую: список-дерево как основной вид + диаграмма как режим «Схема» позже.
- Drag-and-drop по дереву с проверкой недопустимых перемещений (отдел внутрь собственного потомка) — пишите функцию валидации отдельно и покройте тестами, это чистая логика.
- Сразу решите, как выглядит структура на мобильном: диаграммы там не работают, дерево-аккордеон — да.

| 2.3 | Редактор статьи на TipTap: заголовки, списки, таблицы, картинки (мок-загрузка), видео-эмбеды, код-блоки, шаблоны статей | high |
| 2.4 | История версий: список версий, просмотр старой версии, UI отката | medium |

| 2.6 | Поиск по базе: строка поиска, результаты с подсветкой совпадений (поиск по мокам на клиенте) | medium |
| 2.7 | Панель «кто ознакомился» для руководителя: список сотрудников со статусами | low |

**Нюансы этапа:**
- TipTap — правильный выбор, но закладывайте время на кастомизацию: картинки с ресайзом, таблицы и эмбеды потребуют расширений и возни. Не пытайтесь сделать «Notion» — зафиксируйте набор блоков из промта и остановитесь.
- Храните контент как TipTap JSON (не HTML) — это пригодится в Академии для конвертации статьи в урок и для будущей AI-генерации.
- UI доступов делайте, но честную проверку наследования прав не реализуйте на клиенте всерьёз — это ответственность бэкенда. На фронте достаточно функции-заглушки `canView(user, section)` над моками.
- Diff между версиями (красное/зелёное) — заманчиво, но дорого; в MVP достаточно просмотра старых версий целиком.

| 3.5 | Повторяющиеся задачи и назначение на должность: UI настройки повторения, выбор «исполнитель: человек или должность» | low |

**Нюансы этапа:**
- dnd-kit для канбана: продумайте optimistic-обновления (карточка перемещается мгновенно, «сервер» подтверждает после). С TanStack Query это стандартный паттерн — заложите его сразу.
- Карточка задачи — самый насыщенный компонент проекта. Сделайте её сайдпанелью (а не модалкой): удобнее для длинных комментариев и оставляет контекст доски видимым.
- Календарь не пишите сами — возьмите FullCalendar или соберите простой месячный вид; недельные/дневные сетки в MVP не нужны.
- Для комментариев переиспользуйте упрощённый TipTap из этапа 2 — не плодите второй редактор.

| # | Задача | Сложность |
|---|--------|-----------|
| 4.1 | Каталог курсов: карточки курсов, статусы (черновик/опубликован), мои назначенные курсы с прогрессом | low |
| 4.2 | Конструктор курса: структура разделы → уроки, drag-and-drop порядка, настройки курса (последовательное прохождение, дедлайны) | high |
| 4.3 | Редактор урока + импорт из БЗ: контент урока (TipTap), модалка выбора статьи с режимом «ссылка / копия», индикация связи с источником | high |
| 4.4 | Конструктор теста: вопросы (одиночный/множественный выбор, открытый ответ), проходной балл, число попыток | medium |

| 4.6 | Назначение курса: модалка выбора (человек / должность / отдел / внешняя инвайт-ссылка) | medium |
| 4.7 | Дашборд прогресса для руководителя: таблица по сотрудникам, % прохождения, результаты тестов, просроченные, кнопка экспорта CSV | medium |
| 4.8 | Внешний режим партнёра: отдельный минимальный layout (без сайдбара компании) — только назначенные курсы и плеер | medium |
| 4.9 | Сертификат: страница/PDF-вид с именем, курсом, датой (генерация на клиенте) | low |

**Нюансы этапа:**
- Режимы «ссылка / копия» при импорте статьи — ядро продукта, продумайте UX различия: у «ссылки» показывайте бейдж «синхронизировано с базой знаний» и запрет редактирования контента внутри урока; у «копии» — пометку «отвязано от источника». Пользователь должен без документации понять разницу.
- Плеер курса проектируйте mobile-first — сотрудники и партнёры будут проходить курсы с телефона чаще, чем работать с задачами.
- Партнёрский режим — отдельный layout и отдельные роуты (`/learn/...`). Не пытайтесь «прятать» элементы основного интерфейса условиями — обязательно что-то протечёт.
- Открытые ответы в тестах требуют интерфейса ручной проверки для автора курса — не забудьте этот экран, его легко упустить.
- AI-генерацию курсов в этот этап не включаю сознательно — это фаза 2 и завязана на бэкенд/LLM API. Но в конструкторе оставьте место под кнопку «Сгенерировать черновик».

## Рекомендуемый порядок и логика

Последовательность 0 → 1 → 2 → 3 → 4 → 5 не случайна: оргструктура нужна всем остальным модулям (исполнители, доступы, назначения), база знаний нужна Академии (импорт статей), а редактор из этапа 2 переиспользуется в задачах и уроках. Менять местами этапы 2 и 3 можно, 4 раньше 2 — нельзя.
