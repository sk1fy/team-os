# Edler.Pro Academy — второй проход

Дата: 21 июля 2026
Фокус: прогресс, resume, задания, сертификация и переходы между шагами
Связанный обзор: `competitive-research-edler-academy.md`

## Executive summary

Второй проход подтвердил, что основной риск академии Edler.Pro находится не в контенте, а в модели состояния обучения. По отдельности действия работают: ответ выбирается и сохраняется, уроки разблокируются, переходы вперед выполняются. Но агрегированный прогресс, текущий урок и CTA «Продолжить» расходятся между собой.

Критичный наблюдаемый сценарий:

1. На дашборде показано «3 из 6 уроков».
2. Незавершенным является урок «Черкашов Александр».
3. CTA «Продолжить» открывает предыдущий урок «Чат Партнеров».
4. После повторного клика «Продолжить» открывается нужный урок.
5. После прохождения еще двух экранов и входа в сертификацию дашборд показывает «5 из 6».
6. Одновременно следующий шаг «Рынок, Клиенты» уже показывает «1 из 8».
7. Повторный вход в первый шаг открывает «Мой подход» — уже пройденный урок с сохраненным ответом.

Итог: система, вероятно, смешивает как минимум три понятия — «открыт», «посещен» и «завершен» — но показывает их пользователю одним счетчиком. Это подрывает доверие к прогрессу и добавляет лишние действия при каждом возвращении.

## Цель второго прохода

Первый отчет описывал общую архитектуру, UX, mobile и accessibility. Второй проход отвечает на более узкие вопросы:

- куда ведет CTA «Продолжить» после выхода из курса;
- когда урок считается завершенным;
- сохраняются ли ответы;
- как работает блокировка следующего урока;
- что происходит в конце шага;
- синхронизированы ли шаги между собой;
- чем учебная сертификация отличается от итогового сертификата.

## Пройденный сценарий

Во время второго прохода был проверен путь:

`Дашборд → Чат Партнеров → Черкашов Александр → Мой подход → Готово! → Сертификационный лист → Личный кабинет → повторный вход в шаг`

В заданиях выбирались ответы, прямо подтвержденные текстом уроков. Внешние ссылки и коммуникации не отправлялись.

## 1. Наблюдаемая модель курса

### Иерархия

Фактическая структура имеет четыре уровня:

`Уровень партнера → программа → шаг → урок/экран`

Внутри первого шага «СТАРТ!» наблюдается еще одна группировка:

- группа «СТАРТ!»;
  - «Форматы работы»;
  - «Чат Партнеров»;
- группа «Черкашов А.С. Подход к делу»;
  - «Черкашов Александр»;
  - «Мой подход»;
  - «Готово!»;
- группа «Сертификация»;
  - «Сертификационный лист».

То есть отображаемые на дашборде «6 уроков» не равны числу видимых конечных экранов: в дереве показано 2 + 3 + 1 = 6 leaf-уроков, но также присутствуют три группирующих узла. Их интерактивное поведение визуально похоже на поведение уроков, что усложняет понимание и автоматизацию.

### Типы экранов

В проверенном шаге встретились:

- длинный смешанный урок с медиаблоками;
- информационный экран со ссылкой в Telegram;
- длинный текстовый урок с заданием;
- ценностный/установочный урок с заданием;
- короткий success-экран «Готово!»;
- сертификационный лист с двумя заданиями и Google Docs;

Платформа использует одну и ту же оболочку для очень разных сущностей. Это упрощает реализацию, но CTA и критерий завершения не адаптированы к типу экрана.

## 2. Progress и resume

### Найденный дефект: resume открывает предыдущий урок

**Приоритет: P1**

Предусловие: на дашборде первый шаг показывает «3 из 6 уроков», пользователь ранее дошел до урока «Черкашов Александр».

Шаги воспроизведения:

1. Открыть дашборд.
2. В карточке «Шаг 1: СТАРТ!» нажать «Продолжить».

Фактический результат: открывается `/study/3143`, урок «Чат Партнеров» — предыдущий экран.

Ожидаемый результат: открывается `/study/3001`, урок «Черкашов Александр», на котором пользователь остановился.

Влияние:

- пользователь вынужден повторять уже выполненное действие;
- возникает сомнение, сохранился ли прогресс;
- при большом числе коротких уроков трение будет повторяться часто;
- аналитика resume-click не будет отражать фактическое продолжение.

### Найденный дефект: повторный вход открывает завершенный урок

**Приоритет: P1**

После выбора ответа «100%» в уроке «Мой подход», перехода через «Готово!» и сертификационный лист повторный CTA первого шага снова открыл «Мой подход».

При этом выбранный radio-ответ сохранился и после повторного входа был отмечен как checked. Значит, сервер знает о выполненном действии, но resume-алгоритм не использует сохраненное состояние корректно.

Вероятная причина: курс хранит «последний завершенный урок» и открывает его вместо «первого незавершенного» либо счетчик обновляется при открытии следующего экрана, а resume-id — только после отдельного события завершения.

### Счетчик не объясняет семантику

После входа в сертификацию дашборд показал:

- «Шаг 1: СТАРТ!» — 5 из 6;
- «Шаг 2: Рынок, Клиенты» — 1 из 8.

Следующий шаг уже начат, хотя предыдущий формально не завершен. Возможны разные бизнес-правила, но интерфейс их не объясняет.

Нужно явно определить и отображать:

- started — пользователь открыл урок;
- completed — выполнил обязательные действия;
- passed — дал корректные ответы;
- unlocked — получил доступ;
- current — рекомендованный следующий урок.

Один числовой индикатор «N из M» не должен одновременно обозначать эти состояния.

## 3. Задания и валидация

### Что работает

- radio-варианты имеют доступные названия;
- выбранный ответ сохраняется после выхода и повторного входа;
- без ответа переход не выполняется;
- корректный ответ позволяет перейти вперед;
- несколько заданий могут находиться на одном экране;
- после отправки кнопка временно становится disabled, предотвращая повторный submit.

### Проблема: нет явной обратной связи при пропуске

**Приоритет: P1**

При клике «Продолжить» без выбранного варианта переход блокируется. Контейнер задания получает CSS-класс `_error`, но текстовое сообщение не появляется в доступном дереве.

Это color-only feedback. Пользователь внизу длинной страницы может не увидеть ошибочное задание, находящееся выше.

Ожидаемое поведение:

1. Контейнер получает `aria-invalid="true"`.
2. Появляется видимый текст «Выберите один вариант».
3. Сообщение связано с группой через `aria-describedby`.
4. Фокус или viewport переводится к первому ошибочному заданию.
5. Сообщение объявляется через `aria-live`.

### Проблема: проверка знания подменена согласием

В уроке «Мой подход» задание спрашивает, согласен ли пользователь с тезисом автора. Варианты: «100%» и «Нет, я не согласен(а)».

Такое задание проверяет лояльность к формулировке, а не понимание материала. Если второй вариант блокирует обучение, механика становится принудительным согласием.

Лучший вариант:

- заменить на ситуационный вопрос;
- попросить выбрать действие, соответствующее принципу;
- после ответа дать объяснение, почему вариант правильный;
- для ценностной декларации использовать acknowledgement, а не тест.

### Проблема: правильный ответ не дает образовательного feedback

После выбора корректного варианта пользователь сразу переходит к следующему экрану. Не показано:

- почему ответ правильный;
- какой фрагмент урока это подтверждает;
- есть ли начисленный прогресс;
- что именно разблокировано.

Минимальный feedback: «Верно — следующий урок открыт» и одно предложение с объяснением.

## 4. Сертификация

### Что на самом деле представляет собой сертификация

Внутри первого шага «Сертификация» — это подготовительный учебный экран, а не выдача сертификата. Он:

- объясняет назначение общей сертификации;
- описывает два пути построения бизнеса — «снизу вверх» и «сверху вниз»;
- задает сроки подготовки: 3–7, 7–14 или более 14 дней;
- ведет в внешний Google Docs «Лист сертификации»;
- содержит два контрольных вопроса;
- возвращает кнопкой «Личный кабинет».

Одновременно на дашборде есть отдельная заблокированная карточка «Сертификат — После обучения». Термины «сертификация», «сертификационный лист» и «сертификат» выглядят как связанные, но их роли не объяснены.

Рекомендуемая терминология:

- **Проверка шага** — локальные контрольные вопросы;
- **Подготовка к аттестации** — чек-лист знаний;
- **Итоговая аттестация** — встреча/экзамен с куратором;
- **Сертификат** — документ после успешной аттестации.

### Внешний Google Docs — риск версии и доступа

Сертификационный лист вынесен в Google Docs. Это позволяет быстро менять документ, но создает риски:

- недоступность документа из корпоративной сети;
- изменение материала без версионирования курса;
- разрыв аналитики;
- потеря контекста и позиции;
- различия в доступах;
- отсутствие гарантии mobile-friendly представления.

Рекомендация: хранить каноническую версию внутри академии, а внешний документ предлагать только как скачиваемую/печатную копию с номером версии.

### Выход из сертификации не подтверждает завершение

После выбора двух правильных ответов кнопка называется «Личный кабинет», а не «Завершить проверку». После возврата счетчик остается «5 из 6».

Пользователь не понимает:

- были ли ответы отправлены;
- пройден ли экран;
- почему осталось 5/6;
- нужно ли еще что-то сделать;
- почему следующий шаг уже начат.

**Рекомендация:** разделить действия:

- primary: «Проверить ответы»;
- success-state: «Проверка шага завершена»;
- secondary: «Вернуться в академию»;
- на дашборде: явный статус «Шаг завершен» или конкретное незавершенное требование.

## 5. Наблюдаемая state machine

### Текущее поведение

| Событие | Видимый эффект | Проблема |
|---|---|---|
| Открытие первого доступного урока | Следующий остается заблокирован | Нормально |
| Клик «Продолжить» без ответа | URL не меняется, контейнер `_error` | Нет понятного сообщения |
| Выбор верного ответа + «Продолжить» | Кнопка disabled, затем следующий URL | Нет success-feedback |
| Переход на следующий урок | Следующий узел разблокируется | Нормально |
| Возврат на дашборд | Счетчик увеличивается | Неясно, что именно считается |
| CTA «Продолжить» | Открывает предыдущий урок | Ошибка resume |
| Повторный вход | Ответ radio сохранен | Хранение ответа работает |
| Выход из сертификационного листа | Возврат на дашборд | Нет подтверждения завершения |
| Открытие следующего шага | Новый шаг показывает 1/8 | Предыдущий остается 5/6 |

### Рекомендуемая state machine

Для каждого урока:

`locked → available → in_progress → submitted → completed`

Дополнительное состояние для проверяемых заданий:

`submitted → passed | needs_retry`

Правила:

1. `available` не увеличивает progress.
2. `in_progress` сохраняет позицию, но не считается завершением.
3. `completed` выставляется только после успешного server response.
4. Следующий урок разблокируется атомарно с завершением предыдущего.
5. `resumeLessonId = первый available/in_progress урок после последнего completed`.
6. Если шаг завершен, resume ведет на первый незавершенный урок следующего шага.
7. Дашборд получает агрегат из того же источника, что и дерево уроков.

## 7. Регрессионный набор для команды

### Progress/resume

- Новый пользователь: CTA открывает первый урок.
- Открыт, но не завершен урок: CTA возвращает на него.
- Урок завершен: CTA открывает следующий, а не предыдущий.
- Шаг завершен: CTA открывает следующий шаг.
- Все обучение завершено: CTA заменяется на «Посмотреть результаты».
- Два устройства: прогресс и resume-id совпадают.
- Reload во время submit: нет двойного прогресса.
- Back после завершения: завершенный статус не откатывается.

### Задания

- Ни один ответ не выбран.
- Выбран правильный ответ.
- Выбран неправильный ответ.
- На странице несколько обязательных вопросов.
- Ответ изменен перед submit.
- Double-click по submit.
- Потеря сети во время submit.
- Повторный вход после успешного ответа.
- Только клавиатура и screen reader.

### Сертификация

- Все ответы верны.
- Один ответ неверен.
- Внешний документ недоступен.
- Возврат из Google Docs.
- Смена версии чек-листа.
- Переход в следующий шаг до формального завершения предыдущего.
- Повторная аттестация и срок ожидания.

## 8. Рекомендации по аналитике

Нужны отдельные события:

- `academy_resume_clicked` с `expected_lesson_id` и `opened_lesson_id`;
- `lesson_started`;
- `lesson_position_saved`;
- `assignment_validation_failed`;
- `assignment_submitted`;
- `assignment_passed` / `assignment_needs_retry`;
- `lesson_completed`;
- `step_completed`;
- `certification_checklist_opened`;
- `dashboard_progress_rendered`.

Автоматический quality-signal: доля событий, где `expected_lesson_id !== opened_lesson_id`. Для корректной системы она должна быть близка к нулю.

## 9. Предлагаемый UI-контракт

### Дашборд

Карточка шага должна показывать:

- `3/6 завершено`, а не просто `3 из 6`;
- название следующего урока;
- оценку времени;
- статус шага;
- один CTA `Продолжить: Черкашов Александр`;
- причину блокировки, если CTA недоступен.

### Урок

Header:

- breadcrumb `Базовая информация / СТАРТ! / Черкашов Александр`;
- `Урок 3 из 6`;
- статус `В процессе`;
- оценка времени.

Footer:

- secondary `Назад`;
- primary `Завершить и перейти дальше`;
- при задании — `Проверить ответ`;
- после успеха — короткий confirmation и preview следующего урока.

### Сертификация

- отдельный тип экрана с критериями допуска;
- checklist внутри продукта;
- версия материалов;
- явный submit;
- результат проверки;
- история попыток;
- CTA к следующему конкретному действию.

## Итог второго прохода

Edler.Pro удачно выстраивает обязательную последовательность и сохраняет ответы, но не дает пользователю надежной модели «где я сейчас». Три источника истины — дерево уроков, счетчик на дашборде и resume-CTA — расходятся.

Главная рекомендация: до дальнейшего расширения академии стабилизировать state machine и сделать единый серверный контракт прогресса. При 197 видимых уроках даже один лишний возврат на предыдущий экран превращается в системное трение, а неоднозначная сертификация создает риск незавершения курса при фактически выполненных действиях.

## 10. Update после прохождения шагов 2–4

После второго отчета были полностью пройдены шаги «Рынок, Клиенты», «Мой бизнес» и «Услуги Партнера». Это позволило уточнить семантику прогресса и воспроизвести еще два дефекта состояния.

### Уточнение прежнего вывода о `1 из N`

Ранее сочетание `Шаг 1 — 5 из 6` и `Шаг 2 — 1 из 8` было интерпретировано как одновременное начало следующего шага до завершения предыдущего. Последующий проход показал более точную механику:

- после завершения шага 1 шаг 2 оставался `1 из 8`;
- после завершения шага 2 шаг 3 автоматически стал `1 из 8`;
- после завершения шага 3 шаг 4 автоматически стал `1 из 7`;
- после завершения шага 4 шаг 5 автоматически стал `1 из 8`.

Следовательно, счетчик отражает количество доступных/разблокированных уроков, а не только завершенных. UI не сообщает эту семантику и выглядит как стандартный completion progress.

Рекомендуемый контракт:

```text
Завершено: 0 из 8
Доступно сейчас: 1 урок
Следующий урок: [название]
```

### ED-A-11 — race condition при autosave ответов

**Приоритет: P1**

На сертификационном листе два radio-ответа были выбраны подряд. После выхода и повторного входа:

- первый ответ был сброшен;
- второй сохранился.

После повторного выбора первого ответа отдельно, ожидания сохранения и возврата на экран оба значения присутствовали. Только после этого дашборд изменился с `5 из 6` на `6 из 6` и переход к шагу 2 начал работать.

Вероятная причина: каждый change отправляет полный набор ответов, а параллельные запросы завершаются в другом порядке и перезаписывают состояние.

Regression test:

1. На экране с двумя или более вопросами быстро выбрать ответы один за другим.
2. Дождаться окончания запросов.
3. Перезагрузить страницу и повторно открыть урок.
4. Проверить сохранность всех ответов.
5. Повторить с искусственной задержкой и перестановкой response order.

Техническая рекомендация:

- PATCH отдельного `questionId`, а не всего массива;
- optimistic update с version/revision;
- очередь autosave;
- отмена устаревших запросов;
- индикатор pending/saved;
- запрет выхода, пока есть pending writes.

### ED-A-12 — reload текущего урока меняет маршрут

**Приоритет: P1**

Перезагрузка `/study/2966` на сертификационном листе привела к `/study/3009` — предыдущему экрану «Готово!». После этого для возврата к сертификации потребовался еще один переход вперед.

Это отдельный сценарий от CTA resume: даже точный текущий URL не является восстанавливаемым deep link.

Ожидаемый контракт:

- reload сохраняет текущий lesson ID;
- закрытая вкладка восстанавливает тот же урок;
- если доступ изменился, пользователь получает объяснение и ближайший разрешенный маршрут;
- серверный route guard не заменяет разрешенный текущий урок значением stale resume-id.

### Что подтвердилось на шагах 2–4

| Проверка | Результат |
|---|---|
| Завершение шага разблокирует следующий | Да |
| Финальный экран шага возвращает на дашборд | Да |
| Дашборд обновляет `N из M` после возврата | Да |
| Checkbox с несколькими правильными ответами | Работает при последовательном выборе |
| Шесть case-based вопросов на одном экране | Работают, но нет локального progress |
| Radio/checkbox сохраняются автоматически | Да, но быстрые запросы могут конфликтовать |
| Неверно/неполно заполненный экран блокирует переход | Да |
| Resume после повторного входа стабилен | Нет |

### Расширенная state machine

Для корректного описания наблюдаемого поведения нужен отдельный статус доступности:

```text
locked
  ↓ prerequisite completed
available
  ↓ lesson opened
in_progress
  ↓ required answers saved
ready_to_complete
  ↓ completion acknowledged by server
completed
```

Для шага:

```text
locked → available → in_progress → completed
```

`availableLessonCount` не должен использоваться как `completedLessonCount`. На дашборде нужны оба значения либо только завершение плюс название следующего доступного урока.
