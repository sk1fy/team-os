# Исследование academia.rkrs.ru для академии TeamOS

Дата исследования: 21 июля 2026
Фокус: дизайн, структура и пользовательский сценарий прохождения курса
Целевой контекст: модуль «Академия» в TeamOS

## Краткий вывод

`academia.rkrs.ru` — компактный линейный course player для одного партнерского курса. Его сильная сторона не в визуальной уникальности, а в дисциплинированной структуре прохождения:

- короткий вход;
- отдельный welcome-экран;
- полноэкранный учебный режим;
- постоянная программа курса слева;
- реальный общий процент;
- последовательная разблокировка;
- один контрольный вопрос в конце урока;
- понятный feedback при ошибке;
- отдельный CTA перехода вперед.

Для TeamOS стоит перенять именно interaction model. Копировать синий визуальный слой не нужно: существующая дизайн-система TeamOS уже задает более характерный теплый teal, светло-серый фон приложения, карточки 16 px, Radix-компоненты и dark mode.

Наиболее подходящая основа в текущем коде TeamOS — полноэкранный player `AcademyGrokLearnPage`, а не упрощенный `LearnPage`: там уже есть sticky top bar, progress bar, секции, collapsible sidebar, mobile drawer, URL урока и состояние завершения.

## Методика

Проверены:

1. Стартовая форма.
2. Welcome-экран курса.
3. Desktop course player.
4. Структура программы и блокировки.
5. Правильный ответ и успешное завершение урока.
6. Неправильный ответ, объяснение и повторная попытка.
7. Переход на следующий урок.
8. Mobile viewport 390×844 и отсутствие горизонтального overflow.
9. Семантика DOM и доступность контролов.
10. Сопоставление с дизайном и реализациями академии TeamOS.

Для входа использовались синтетические тестовые данные. Внешние сообщения и ссылки не отправлялись.

## 1. Пользовательский путь

### 1.1 Стартовая форма

Первый экран максимально узкий:

- надпись `RAKURS · КУРС ДЛЯ ПАРТНЁРОВ`;
- пояснение «Введите имя и телефон, чтобы начать обучение»;
- поле имени;
- маскированное поле телефона;
- disabled-кнопка «Начать обучение» до валидного заполнения.

Композиция — одна карточка по центру большого белого поля. Она хорошо удерживает фокус и не перегружает пользователя до начала курса.

Недостаток: рядом с отправкой телефона не видно текста о цели обработки данных, политики конфиденциальности или согласия. Для корпоративного TeamOS такой gate обычно не нужен: пользователь уже авторизован, а имя и рабочие контакты известны системе.

### 1.2 Welcome-экран

После входа показывается отдельная вводная карточка:

- автор и должность;
- маркер «Курс для партнёров»;
- название курса;
- короткое социальное доказательство;
- объяснение пользы;
- ссылка на поддержку;
- один primary CTA «Начать обучение».

Это хороший transition layer между каталогом и первым уроком. Пользователь понимает, кто автор, зачем проходить курс и что будет результатом.

Для TeamOS welcome стоит сделать частью карточки курса/экрана запуска, но использовать данные сотрудника и компании:

- автор или владелец курса;
- кому назначен курс;
- цель;
- продолжительность;
- число уроков и тестов;
- дедлайн;
- обязательность;
- результат/сертификат.

### 1.3 Course player

Основной экран состоит из трех устойчивых зон:

```text
┌─────────────────────────────────────────────────────────┐
│ Название курса              5%               Пользователь│
├────────────────┬────────────────────────────────────────┤
│ Программа      │ Раздел · Урок 1.3                      │
│                │ Заголовок урока                        │
│ Раздел 1  1/9  │ Видео / текст / списки                 │
│  ● текущий     │                                        │
│  ✓ пройденный  │ Проверь себя                           │
│  🔒 закрытый   │ ○ вариант ответа                       │
│                │ [Проверить ответ]                      │
│ Раздел 2  0/3  │ [Следующий урок →]                     │
└────────────────┴────────────────────────────────────────┘
```

Header показывает название курса, общий процент и пользователя. Sidebar показывает программу. Main содержит только текущий урок.

Это правильная модель для TeamOS: пользователь остается в учебном контексте и не отвлекается на основную навигацию корпоративной системы.

## 2. Информационная архитектура курса

В курсе 19 уроков в пяти секциях:

1. **Базовая информация** — 9 уроков.
2. **Массовый поиск** — 3 урока.
3. **Фоновый режим** — 2 урока.
4. **Пользовательские сценарии** — 2 урока.
5. **Продажа и демонстрация виджета** — 3 урока.

Структура отражает хороший образовательный порядок:

`Проблема → возможности продукта → настройка → пользовательские кейсы → продажа`

Первый раздел строит ценность продукта и базовую терминологию. Затем курс переходит к конкретным режимам, сценариям использования и коммерческой демонстрации.

### Состояния в программе

Уроки имеют четко различимые состояния:

- текущий — синяя точка и акцентный текст;
- доступный — обычная кнопка;
- закрытый — lock и disabled;
- пройденный — влияет на счетчик секции и общий процент.

В отличие от Edler, `1 / 9 уроков` действительно означает число завершенных уроков. После первого quiz общий прогресс изменился с 0% на 5%, что соответствует 1 из 19.

### Accordion секций

Секции отображаются как раскрываемые группы. Первая раскрыта, остальные свернуты. Это позволяет показать полный масштаб программы, но не выводить одновременно все 19 названий.

Для TeamOS это особенно полезно, потому что корпоративные курсы могут быть длиннее и состоять из регламентов, статей базы знаний, видео и тестов.

## 3. Дизайн-система сайта

### Палитра

Основной интерфейс использует:

- белый фон;
- slate-текст (`rgb(51, 65, 85)`);
- почти черные заголовки (`rgb(15, 23, 42)`);
- primary blue `#2563EB`;
- светло-синие active-состояния;
- серые границы и disabled-состояния.

### Типографика

- Inter/system-ui;
- базовый текст 16/24 px;
- заголовок урока на desktop — 36/45 px;
- ограниченная ширина статьи — 768 px;
- короткие абзацы и списки;
- внутри урока используется `h1`, затем `h3`.

Ширина 768 px комфортна для учебного чтения, хотя иерархию `h1 → h3` лучше исправить на последовательную.

### Геометрия

- sidebar: 288 px;
- article: до 768 px;
- primary controls: radius 12 px;
- welcome CTA: 448 px, radius 12 px;
- интерфейс преимущественно плоский, без тяжелых карточек;
- header sticky;
- main layout центрирован.

### Визуальный характер

Дизайн нейтральный, чистый и утилитарный. Он выглядит как отдельный microsite, а не как часть большой корпоративной системы. Это подходит внешнему партнерскому курсу, но для TeamOS нужен более плотный системный контекст: назначение, дедлайн, роль сотрудника, связанные регламенты и прогресс команды.

## 4. Учебный UX

### Правильный ответ

После выбора корректного варианта:

- урок сразу засчитывается;
- в header обновляется общий процент;
- счетчик секции меняется с `0 / 9` на `1 / 9`;
- следующий урок разблокируется;
- появляется текст «Квиз пройден»;
- CTA «Следующий урок» становится активным.

Это хороший атомарный переход: локальный результат, агрегированный progress и доступность следующего урока обновляются одновременно.

### Неправильный ответ

После ошибочного ответа:

- варианты блокируются;
- показывается конкретное объяснение;
- появляется CTA «Попробовать ещё раз»;
- следующий урок остается закрытым.

Пример feedback не просто сообщает «неверно», а объясняет, почему нужная возможность продукта решает описанный кейс. Это один из лучших паттернов сайта и его обязательно стоит перенять в TeamOS.

### Sequential learning

Следующий урок недоступен, пока не пройден текущий quiz. Sidebar при этом заранее показывает названия закрытых уроков. Пользователь понимает маршрут, но не может пропустить обязательную проверку.

### Контент урока

Проверенные уроки используют:

- крупный заголовок;
- breadcrumb секции и номера;
- видео в формате 16:9;
- короткое вступление;
- смысловые подзаголовки;
- маркированные списки;
- один contextual quiz;
- CTA вперед.

Контент значительно компактнее уроков Edler и лучше соответствует модели microlearning.

## 5. Mobile и адаптивность

На 390×844 DOM перестраивается корректно:

- ширина документа равна viewport — 390 px;
- горизонтального overflow нет;
- article занимает полную ширину;
- появляется кнопка «Открыть меню»;
- программа превращается в drawer шириной 288 px;
- есть отдельная кнопка закрытия;
- header сохраняет название и процент;
- имя пользователя скрывается, освобождая место.

Это структура, которую стоит сохранить в TeamOS. В текущем `AcademyGrokLearnPage` уже реализована почти такая же модель: mobile drawer, overlay, sticky header и отдельный progress bar для малых экранов.

Важно: при открытии mobile drawer должен автоматически блокироваться scroll контента, фокус должен переходить в меню и возвращаться на trigger после закрытия.

## 6. Доступность

На проверенном lesson-экране:

- присутствуют `main`, `aside` и два `nav`;
- все 19 кнопок имеют доступные имена;
- radio имеют содержательные accessible names;
- disabled-состояния доступны семантически;
- кнопка мобильного меню имеет имя;
- feedback ошибочного ответа присутствует в DOM.

Недочеты:

- заголовки перескакивают с `h1` на `h3`;
- не подтвержден `aria-live` для динамического feedback;
- не видно явной `fieldset/legend`-семантики quiz-группы;
- disabled-уроки не объясняют причину блокировки;
- процент передан как отдельный generic text без описания «Прогресс курса».

## 7. Найденные проблемы

### P1 — сломано изображение автора

На welcome-экране фотография автора не загружается. Проверка показала `naturalWidth = 0`, источник ведет на `/src/assets/images/author-maxim.jpeg`.

Это похоже на production-ссылку на исходник вместо собранного asset URL.

**Влияние:** первый брендовый экран выглядит незавершенным, вместо фотографии показывается alt-текст.

### P1 — сбор телефона без видимого consent-контекста

Стартовая форма требует имя и телефон, но на экране не видно:

- ссылки на политику;
- цели обработки;
- обязательности полей;
- согласия;
- информации о том, куда будет передан номер.

Для TeamOS этот экран не нужен внутри авторизованного продукта. Для публичного курса следует добавить privacy copy и минимизировать собираемые данные.

### P2 — нумерация URL/урока не совпадает с порядком

После первого урока `/lesson/s1/1.1` следующий видимый урок открылся как `/lesson/s1/1.3`, а breadcrumb показал «Урок 1.3». В sidebar это второй урок секции.

Если `1.2` удален, внутренний ID не должен использоваться как пользовательская нумерация. Нужно показывать вычисляемое `Урок 2 из 19` или `1.2` по текущему порядку.

### P2 — нет оценки времени

В sidebar и header нет:

- общей длительности;
- длительности урока;
- прогноза окончания;
- числа вопросов.

Процент помогает, но не отвечает на вопрос «успею ли я пройти следующий урок сейчас».

### P2 — успешный feedback слабее ошибочного

При ошибке пользователь получает содержательное объяснение. При успехе видит только «Квиз пройден». Стоит дать короткое подтверждение ключевого знания и следующий контекст.

### P2 — причина блокировки скрыта

Закрытые уроки имеют lock и disabled, но hover/focus explanation не видно. Tooltip «Пройдите предыдущий урок» снизит неопределенность.

### P2 — отсутствует явный выход из player

В проверенном header название курса реализовано как button, но назначение неочевидно. Для TeamOS нужен явный icon-button `X` с accessible name «Закрыть курс» и предупреждение только при несохраненном ответе.

### P3 — минимальный контекст пользователя

Показывается только имя. Для TeamOS полезнее отображать дедлайн/назначение, а профиль оставить в основном приложении.

## 8. Что перенести в TeamOS

### Перенести почти без изменений

1. Full-screen course player вне основного app shell.
2. Sticky header с названием и общим progress.
3. Sidebar 280–288 px с accordion-секциями.
4. Состояния current/completed/locked.
5. Компактную ширину статьи 720–780 px.
6. Один contextual quiz после смыслового блока.
7. Explanation feedback для неправильного ответа.
8. Атомарное обновление progress + unlock + CTA.
9. Mobile drawer вместо постоянного sidebar.
10. Welcome перед первым запуском курса.

### Адаптировать под TeamOS

- blue заменить на TeamOS primary teal;
- белый microsite-фон заменить на `bg-page`/`bg-surface-muted`;
- использовать `rounded-xl`, `shadow-card`, границы TeamOS;
- добавить dark mode;
- показывать дедлайн, назначившего руководителя и обязательность;
- поддержать связанные статьи базы знаний;
- показывать ручную проверку open-вопросов;
- использовать toast только как дополнительный feedback, не вместо inline-state;
- сохранять текущий урок в URL;
- добавить exit/dismiss guard для несохраненного ответа.

## 9. Предлагаемый дизайн TeamOS Academy Player

### Header

Высота: 56–60 px, sticky.

Слева:

- mobile trigger программы;
- desktop collapse-sidebar;
- название курса, truncate;
- необязательный badge «Обязательный».

По центру/справа:

- `Урок 4 из 19`;
- процент;
- progress bar;
- кнопка закрытия player.

Использовать:

- primary-600 `#2F7E78` для активных элементов;
- primary-50 для active row;
- success-600 для completed;
- slate-300 для locked.

### Sidebar

Desktop: 280 px, карточка `rounded-xl border bg-surface shadow-card`.

Секция:

- название;
- `3/9 завершено`;
- тонкий progress bar;
- chevron;
- поддержка collapse.

Урок:

- номер;
- название до двух строк, не однострочный truncate;
- длительность/тип (`Видео · 6 мин`, `Тест · 3 вопроса`);
- status icon;
- tooltip причины lock.

Mobile:

- drawer 88–92% ширины, максимум 320 px;
- overlay;
- focus trap;
- scroll lock;
- автоматическое закрытие после выбора урока.

### Article

- максимальная ширина контента 760 px;
- карточка `rounded-xl border bg-surface`;
- padding 20 px mobile / 32 px desktop;
- label `Раздел 1 · Урок 2 из 19 · 7 минут`;
- `h1` 24 px в TeamOS, а не 36 px competitor-style;
- line-height 1.6;
- видео 16:9 с poster, captions и transcript;
- sticky или финальный footer действий.

### Quiz

Карточка на `surface-muted`:

- heading «Проверьте себя»;
- номер вопроса;
- prompt;
- radio/checkbox через компоненты TeamOS;
- inline validation;
- primary «Проверить ответ»;
- feedback с icon, heading и объяснением;
- после успеха primary меняется на «Завершить и продолжить».

Не стоит автоматически отправлять ответ сразу по выбору: отдельная кнопка дает пользователю возможность проверить решение и снижает случайные ошибки.

### Footer урока

```text
[← Предыдущий]          Сохранено     [Завершить и продолжить →]
```

На mobile действия становятся вертикальными или primary занимает всю ширину.

## 10. Сопоставление с текущим TeamOS

### Базовый `LearnPage`

Сильные стороны:

- уже есть двухколоночный layout;
- TeamOS tokens;
- locked/completed states;
- quiz-карточка;
- error/loading states.

Пробелы относительно исследованного сайта:

- нет секций в sidebar;
- нет общего progress в header;
- нет mobile drawer;
- текущий урок хранится только в state, а не в URL;
- quiz фактически завершается без локального scoring feedback;
- нет next-step success state;
- названия уроков принудительно truncate в одну строку.

### `AcademyGrokLearnPage`

Это более подходящая основа. Уже реализованы:

- sticky top bar;
- общий процент и progress bar;
- скрываемая программа;
- sections;
- mobile drawer и overlay;
- URL `?lesson=`;
- current/completed/locked states;
- отдельный close-player;
- toast завершения;
- более плотная карточная геометрия.

Рекомендуемый путь: унифицировать основную академию вокруг Grok-player и добавить в него лучшие паттерны `academia.rkrs.ru` — question-level validation, правильный/неправильный feedback, retries и атомарный unlock.

## 11. Компонентная карта TeamOS

| Зона | Использовать в TeamOS | Дополнение |
|---|---|---|
| Header | `Button`, `Badge`, `ProgressBar` | `CoursePlayerHeader` |
| Sidebar | `Button`, `Tooltip` | `CourseOutline`, `CourseSection` |
| Mobile menu | существующий responsive aside | focus trap + scroll lock |
| Article | `RichTextView` | metadata row, duration |
| Video | TipTap video embed | captions, transcript, poster |
| Quiz | `RadioGroup`, `Checkbox`, `Textarea` | scoring feedback, retries |
| Feedback | `Toast` + inline callout | inline state обязателен |
| Footer | `Button` | autosave status + prev/next |
| Exit | icon button | `dismissGuard` при pending input |

## Финальная рекомендация

Лучший вариант для TeamOS — гибрид:

- структура и feedback `academia.rkrs.ru`;
- teal-дизайн, компоненты, dark mode и corporate context TeamOS;
- секционный полноэкранный player из `AcademyGrokLearnPage`;
- live-связи с базой знаний;
- надежная state machine прогресса с lesson-level URL;
- прозрачная оценка времени, дедлайн и обязательность.

Такой player будет выглядеть частью TeamOS, но сохранит главное достоинство референса: пользователь всегда понимает, где находится, что уже завершено, почему следующий урок закрыт и какое действие нужно выполнить сейчас.
