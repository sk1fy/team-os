# План frontend-QA проверки TeamOS

Инструкция для агента-исполнителя. Выполнять секции по порядку приоритета: P0 → P1 → P2. Каждый найденный дефект фиксировать в формате из раздела «Формат репорта».

---

## 0. Окружение и особенности приложения (прочитать перед началом)

### Запуск

```sh
npm run dev   # Vite dev server, http://localhost:5173
```

### Критично знать перед тестированием

1. **Бэкенда по умолчанию нет.** Mock API (`src/api/client.ts`) добавляет задержку **300–500 мс** и **5% случайных ошибок** на каждый запрос. TanStack Query настроен на `retry: 2`, поэтому случайная ошибка обычно «съедается» ретраями. Ошибка, которую вы увидите один раз и не сможете воспроизвести, может быть этой искусственной ошибкой — **прежде чем заводить баг про «ошибку сети», повторите действие 3–5 раз**.
2. **Данные in-memory** (`src/api/fixtures.ts`): всё, что вы создали/изменили, живёт до перезагрузки страницы. **F5 сбрасывает базу к фикстурам.** Это ожидаемое поведение, а не баг. Но UI-состояние из Zustand (тема, сайдбар, фильтр по отделу — ключ `teamos-ui` в localStorage) обязано переживать перезагрузку.
3. **Авторизация в mock-режиме фиктивная**: текущий пользователь — всегда владелец компании (`owner`). Формы логина/регистрации существуют и должны валидироваться, но реальной сессии нет.
4. Есть HTTP-режим по модулям через env `VITE_API_MODE_<MODULE>=http` (`VITE_API_URL`, по умолчанию `http://localhost:8080/api/v1`). Если бэкенд не поднят — тестируем только mock-режим, HTTP-секции пропускаем с пометкой «skipped: no backend».
5. **Весь UI на русском.** Английский текст, «undefined», «NaN», непереведённые ключи — дефект.
6. Для детерминированной проверки ошибок допустимо **временно** поменять `ERROR_RATE` в `src/api/client.ts` на `1` (все запросы падают) или `0` (ничего не падает). После проверки обязательно вернуть `0.05` и не коммитить.

## 1. [P0] Неправильный ввод в формах

Прогнать на каждой форме: **логин, регистрация, инвайт, создание/редактирование сотрудника (`AddUserModal`, `EmployeeEditModal`), создание статьи БЗ, создание задачи/колонки, группа распределения, график смен**.

Набор значений для каждого текстового поля:

| # | Ввод | Ожидание |
|---|---|---|
| 1 | Пустое значение → submit | Ошибка у поля, форма не отправляется, фокус/скролл к первой ошибке |
| 2 | Только пробелы `"   "` | Считается пустым (trim), не создаётся сущность с пустым именем |
| 3 | Пробелы в начале/конце `"  Иван  "` | Значение сохраняется без крайних пробелов ИЛИ поведение консистентно везде |
| 4 | 300+ символов (сгенерировать строку) | Ограничение длины или корректный перенос; таблицы/карточки/сайдбар не разъезжаются после сохранения |
| 5 | Строка без пробелов из 100 символов (`WWW...`) | Нет горизонтального скролла страницы, обрезка/`word-break` |
| 6 | Спецсимволы `<script>alert(1)</script>`, `"><img src=x onerror=alert(1)>`, `'; DROP TABLE --` | Отображаются как текст (экранирование), скрипт НЕ выполняется — проверить и в таблице, и в карточке, и в тосте |
| 7 | Эмодзи `😀👨‍👩‍👧‍👦🇷🇺`, RTL-текст `مرحبا` | Не ломают вёрстку, сохраняются и отображаются |
| 8 | Email: `a`, `a@`, `a@b`, `почта@домен.рф`, `a b@c.d`, `A@B.COM` | Невалидные отклоняются с русским сообщением; кириллический домен и верхний регистр — зафиксировать фактическое поведение |
| 9 | Телефон (если есть поле): буквы, `+7 (999) 000-00-00`, `89990000000`, 5 цифр | Единый формат или маска; невалидное — ошибка |
| 10 | Дата (график смен, дедлайны): прошлая дата, 30 февраля, ручной ввод мусора | Невалидные даты невозможно ввести или есть ошибка |

Особое внимание:
- **TipTap-редактор** (БЗ, уроки): вставка (`Ctrl+V`) большого текста, вставка HTML со стилями из буфера, пустая статья — сохранение и последующий рендер в read-only виде и на `/share/article/:articleId`.
- Поиск/фильтры (сотрудники, БЗ): спецсимволы и regex-метасимволы `(`, `[`, `*`, `\` в строке поиска не должны ронять страницу (типичный краш — непроэкранированный `new RegExp(query)`).

## 2. [P0] Состояния: загрузка, ошибки, пустота

Благодаря mock API все состояния достижимы честно.

- **Загрузка**: на каждой странице при первом заходе виден skeleton/spinner (задержка 300–500 мс это гарантирует). Нет «прыжка» layout после загрузки (CLS), нет мигания пустого состояния до данных.
- **Ошибка запроса**: временно `ERROR_RATE = 1` → каждая страница должна показать `ErrorState` с русским текстом и кнопкой повтора, а не белый экран и не бесконечный спиннер. Проверить: кнопка «Повторить» реально перезапрашивает (после возврата `ERROR_RATE = 0.05` данные приходят).
- **Ошибка мутации**: при `ERROR_RATE = 1` попытаться создать сотрудника/задачу → тост с ошибкой, модалка НЕ закрывается (введённые данные не теряются), сущность не появляется в списке.
- **Пустые списки**: удалить все элементы (или найти фильтр без результатов) на каждой странице → `EmptyState` с осмысленным текстом, а не пустая область/`0`/краш. Отдельно: поиск без результатов ≠ «данных нет вообще» — тексты должны различаться.
- **ErrorBoundary**: проверить, что рендер-краш одной страницы (если найдёте) не убивает всё приложение.
- **Несуществующие id**: `/employees/nonexistent-id`, `/learn/fake`, `/share/article/fake`, `/distribution/fake` → страница «не найдено» (ApiError 404 «… не найден»), не белый экран, не вечный спиннер.

## 3. [P1] Вёрстка и адаптивность

Разрешения: **375×812 (mobile), 768×1024 (tablet), 1280×800 (desktop)** + промежуточное ~1024. На каждой странице из карты маршрутов:

- Нет горизонтального скролла body; таблица сотрудников и kanban скроллятся внутри своего контейнера.
- Сайдбар: сворачивание/бургер на мобильном, оверлей закрывается.
- Модалки на mobile: помещаются, контент скроллится внутри, кнопки действий доступны, фон не скроллится под модалкой.
- Календарь графика на 375px — самый рискованный экран, проверить внимательно.
- Zoom браузера 90% / 110% / 150% (Ctrl+±): ничего не наезжает, текст не обрезается, sticky-элементы не перекрывают контент.
- Поворот (эмуляция: 812×375): layout перестраивается, модалки выживают.
- **Тёмная/светлая тема** на каждой странице: нет «белых» островов в тёмной теме, нечитаемого текста, жёстко заданных цветов. Переключение темы применяется мгновенно и переживает F5 (persist `teamos-ui`).

## 4. [P2] Кроссбраузер / кроссплатформа

Основная среда — Chromium. Если Firefox/Safari недоступны в окружении — пометить «not run» и перечислить, что проверять вручную:

- Safari/iOS: положение курсора и зум при фокусе инпута (font-size < 16px вызывает автозум), позиционирование модалок при открытой клавиатуре, `position: sticky`, скролл внутри модалок (`-webkit-overflow-scrolling`), date-инпуты.
- Firefox: скроллбары (ширина отличается), рендер таблиц, dnd.
- Android Chrome: drag-and-drop канбана тачем (dnd-kit требует явных touch-сенсоров — высокий риск, что dnd не работает тачем вообще; проверить эмуляцией touch), клавиатура и модалки.

## 5. [P1] Доступность (быстрый аудит)

- **Tab-обход** каждой страницы: порядок логичен, фокус виден (outline не сброшен глобальным `outline: none`), нет ловушек фокуса вне модалок.
- **Модалки (Radix)**: фокус попадает внутрь при открытии, `Esc` закрывает, фокус возвращается на триггер, `Tab` зациклен внутри, `role="dialog"` + `aria-labelledby`/`aria-label` присутствуют. Проверить и кастомные модалки (`AddUserModal`, `EmployeeEditModal`, `DistributionGroupModal`) — они могут быть не на Radix.
- **Формы**: каждый инпут связан с `<label>` (клик по лейблу фокусирует поле); сообщение об ошибке связано через `aria-describedby` или хотя бы визуально рядом.
- **Табы/дропдауны**: управление стрелками, `aria-selected`.
- Иконки-кнопки (закрыть, удалить, меню «⋯») имеют `aria-label` — кнопка без текста и без label = дефект.
- Контраст: вторичный серый текст на фоне в обеих темах ≥ 4.5:1 (проверить точечно через computed styles: топ-3 подозрительных места на страницу).
- `document.title` меняется по страницам (или зафиксировать, что нет).

## 6. [P0] Формы: валидация и submit

Для каждой формы из раздела 1:

- Момент подсветки ошибки: on blur / on submit / on change — зафиксировать и проверить консистентность между формами.
- Ошибка исчезает после исправления поля (не остаётся «залипшей»).
- Submit с невалидными данными не шлёт запрос (проверить отсутствие мутации: сущность не создалась).
- Кнопка submit во время запроса: **заблокирована + индикатор** (задержка 300–500 мс делает это видимым). После ошибки — разблокируется.
- `Enter` в текстовом поле: отправляет форму один раз (не дублирует, не перезагружает страницу).
- Закрытие модалки крестиком/`Esc`/кликом по фону во время запроса — что происходит? Не должно быть «осиротевшего» результата и краша.

## 7. [P0] Кнопки и повторные действия

- **Double-click** по каждой кнопке создания/удаления (сотрудник, задача, статья, группа, смена): сущность создаётся/удаляется **один раз**. Задержка мока 300–500 мс — идеальное окно для дабл-клика; отсутствие `disabled`/`isPending`-гварда почти гарантированно даст дубль.
- Быстрое повторное нажатие «Пригласить», «Отметить всё прочитанным», переключателей (актив/деактив сотрудника).
- Drag-and-drop канбана: быстро перетащить карточку дважды подряд; перетащить и сразу F5 — состояние консистентно после ретраев/ошибок (при 5% ошибке мутации карточка должна откатиться или показать ошибку, не «раздвоиться»).
- Кнопки в состоянии disabled действительно не кликабельны (в т.ч. через Enter/Space с клавиатуры).

## 8. [P1] Навигация, URL, перезагрузка

- Прямой заход (deep link) на каждый route из карты — страница открывается сама по себе, без предварительной навигации.
- Back/Forward браузера: после открытия профиля сотрудника, курса, группы распределения — «назад» возвращает корректно, состояние списка (фильтры/скролл) — зафиксировать поведение.
- Фильтры/поиск/активная вкладка: сохраняются ли в URL (query params)? Если нет — F5 их сбрасывает; зафиксировать как naблюдение (improvement, не баг).
- F5 на каждой странице: она восстанавливается (с учётом сброса mock-данных). Тема/сайдбар/фильтр отдела — восстановились из localStorage.
- `/structure` → редирект на `/employees` с `replace` (back не зацикливается).
- Редиректы `/auth` → `/auth/login`, `/auth/create-company` → `/auth/register`.
- 404 на мусорный URL, и с него есть ссылка «на главную».
- `/learn/:courseId` (вне AppLayout): выход из плеера возвращает в академию; back из плеера не ломается.

## 9. [P1] Роли и права

В mock-режиме текущий пользователь всегда `owner`, поэтому проверка ограничена, но выполнимая часть:

- Роли: `owner | admin | employee | partner`; статусы: `active | invited | deactivated`. Гварды — в `src/lib/userGuards.ts` и `src/lib/inviteRules.ts` (прочитать, чтобы знать ожидаемые правила).
- Через UI: попытаться изменить роль/статус самого владельца, деактивировать владельца, понизить последнего владельца — UI должен запрещать (кнопка скрыта или disabled с объяснением), а API-гвард — отклонять с русским сообщением. Зафиксировать, **прячется ли контрол или дизейблится** — и консистентно ли это.
- Для полноценной проверки «глазами employee»: временно поменять `CURRENT_USER_ID` в `src/api/fixtures.ts` на id обычного сотрудника, перезагрузить, пройти по всем страницам: скрыты ли админские действия (добавление сотрудника, редактирование чужих профилей, управление группами распределения). Вернуть значение после проверки.
- HTTP-режим (если доступен): убедиться через `read_network_requests`, что при ограниченной роли данные, которые UI прячет, **не приходят в ответах** (скрытие только на фронте = дефект безопасности, severity high).

## 10. [P2] Производительность на практике

- Сгенерировать объём: добавить в фикстуры ~500 сотрудников / ~200 задач в колонке / ~100 статей (временная правка `fixtures.ts`) и проверить: таблица сотрудников (есть ли виртуализация — скорее всего нет), плавность скролла, время фильтрации при вводе в поиск (есть ли debounce — печатать быстро и смотреть на лаги/количество перерендеров).
- Kanban с 200 карточками: перетаскивание не фризит.
- TipTap: статья на ~50 000 знаков — открытие редактора, набор текста без лагов.
- Переключение темы на тяжёлой странице — без секундного фриза.
- В консоли: предупреждения React о key, warning-и, любые ошибки при обычной навигации по всем страницам — ноль ошибок в консоли это требование.
- Повторные переходы между страницами: TanStack Query кэширует — данные показываются мгновенно из кэша с фоновым рефетчем (проверить отсутствие повторного полноэкранного спиннера).

## 11. [P0] CRUD-циклы по модулям (пользовательские операции)

Полный цикл создать → отобразилось в списке → изменить → удалить → исчезло, плюс побочные эффекты:

| Модуль | Сценарий |
|---|---|
| Сотрудники | добавить (invited) → редактировать → деактивировать → активировать; фильтр по отделу (глобальный, из Zustand) влияет на все связанные страницы |
| База знаний | создать статью → редактировать rich-text → открыть `/share/article/:id` → удалить; связь с уроками Академии (`link` — контент живой, `copy` — независимый): изменить статью и проверить, что link-урок обновился, copy-урок — нет |
| Задачи | создать колонку → создать задачу → dnd между колонками → dnd порядка колонок → удалить колонку с задачами (что с задачами?) |
| Академия | пройти урок в `/learn/:courseId` → прогресс отражается в академии и в виджете профиля сотрудника |
| График | назначить смену → изменить → удалить; сотрудники «без графика» отображаются; проверить границы месяца/недели, переход через месяц |
| Распределение | создать группу → добавить участников → изменить веса/правила → удалить группу |
| Уведомления | счётчик непрочитанных в топбаре; прочитать одно/все → счётчик обновился без F5 |

После **каждой** мутации проверять: список обновился без ручного F5 (инвалидация ключей `['module', ...]`), тост показан, счётчики/виджеты на других страницах согласованы.

## Формат репорта (обязателен для каждого дефекта)

```markdown
### [SEV-1..4] Краткий заголовок

- **URL/route:** /employees (модалка «Добавить сотрудника»)
- **Окружение:** Chrome 12x, viewport 375×812, тема тёмная, mock-режим
- **Шаги:**
  1. ...
  2. ...
- **Ожидаемое:** ...
- **Фактическое:** ...
- **Скрин:** (обязателен для визуальных багов)
- **Консоль:** текст ошибок из read_console_messages (или «чисто»)
- **Сеть:** аномалии из read_network_requests, если релевантно (HTTP-режим)
- **Воспроизводимость:** стабильно / N из M попыток (важно из-за 5% mock-ошибок!)
```

Severity: SEV-1 — краш/потеря данных/XSS; SEV-2 — функция не работает; SEV-3 — работает с неудобством/визуальный дефект; SEV-4 — косметика/улучшение.

Итог прогона: сводная таблица «раздел плана → passed/failed/skipped + ссылки на дефекты», список временных правок кода (ERROR_RATE, CURRENT_USER_ID, фикстуры) с подтверждением, что всё откачено (`git diff` чистый).
